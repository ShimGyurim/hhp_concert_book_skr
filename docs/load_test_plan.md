### 시나리오 <1>

사용자가 콘서트 조회  → 좌석 조회 → 사용자 로그인 → 예약 토큰 발급 → 임시 예약 까지 하는 상황

(1000명의 가상 사용자 동시접)

### 선정 사유

위 시나리오가 콘서트 예매 시작 시간에 맞춰 트래픽이 몰리는 상황이기 때문임.

### 부하테스트 종류

peak test (최고부하테스트) :

- 예약은 1회성으로 시도가 가능하기 때문에 load test 는 어려움
- 예약 가능 시작 시점에 사용자들이 몰리기 때문에 peak test 로 함.

### 목표 사항

1000 명의 사용자 동시 접속 시 .각 api 별 1000ms 이내 응답 목표.

- 사용자 당 트랜잭션 예상 갯수 : 콘서트조회(1) + 좌석조회(1) + 사용자로그인 (1) + 예약토큰 발급 (1) + 임시 예약 (1) = 5개
- 응답시간 : 각 사용자당 api 호출 1000ms 내 요청~응답 성공 목표
- 목표 tps: 한 콘서트에 좌석이 50개 * 각 사용자별 사용 api (5개) ⇒ 최소 tps 250
- 동시성이 잘 지켜졌는지 정합성 확인

### 시나리오 <2>

사용자 잔액 충전 → 사용자 잔액 조회  (100명의 가상사용자가 2분동안 2초에 한번 요청)

### 선정 사유

- 잔액충전/조회 기능은 토큰 인증 및 로그인 과정이 필요없기 때문에 반복적으로 http 요청 용이

  → load test 용이.

- 잔액 충전 및 조회기능은 큰 부하로 1분이상 돌려도 서버가 이상없는지 여부 확인

### 부하테스트 종류

- load test (부하 테스트) : 지속적으로 큰 트래픽을 유발함에도 서버가 안정적으로 살아있는지 확인.

### 목표 사항

- 충전 / 조회 api 모두 요청~응답까지 500ms 이내 목표 (threshold 옵션 이용)
- 사용자 당 트랜잭션 갯수 : 충전 (1) + 조회 (1) = 2개
- 목표 tps : 100명이 동시에 충전 및 조회 시도시 2초내로 처리 완료 → 100 tps 이상 목표
